Bug Investigation

The Silent Bug That Broke Equatorial Tracking for Every User

A transposed matrix caused index overflow on every cycle. The tracking loop swallowed the exception. Equatorial drift correction never ran. Three critical bugs hiding in plain sight.
By Bogdan • June 10, 2026 • Branch: fix/tracking-kalman-ekf
3
Critical bugs found

Two filters broken, one matrix routine returning identity for any input larger than 2×2. All three silently disabled core tracking features.

On June 10, 2026, a systematic audit of the HYBRID Telescope Control tracking subsystem uncovered three critical bugs that had been silently disabling equatorial and Alt-Az drift correction since the initial implementation. The tracking loop, designed to catch and continue past exceptions for reliability, had inadvertently hidden the failures from every user, every session, every night of imaging.

The first and most severe bug lived in the Kalman filter's Update method — the mathematical core of the equatorial tracking system. The code applied MatrixVectorMultiply(Transpose(K), y), where K is a 4×2 matrix. Transposing a 4×2 produces a 2×4 matrix. Multiplying that by the 2-element innovation vector y yields a 2-vector. But the state vector is 4 elements. The index overflow threw an exception every cycle.

The Architecture: Three Layers of Tracking

HYBRID's tracking system operates on three layers that run simultaneously. Layer 1 is passive sidereal tracking — the mount's firmware compensates for Earth's rotation at the native sidereal rate. Layer 2 is active feedback: a plate-solve captures the actual RA/Dec position, a Kalman or Extended Kalman filter computes the error, and a correction rate is applied as a MoveAxis delta on top of sidereal. Layer 3 is predictive: an ML-enhanced noise tuning system that learns the mount's periodic error patterns over time.

LAYER 1 — PASSIVE Sidereal / ASCOM firmware tracking (always on) │ ▼ LAYER 2 — ACTIVE Plate-solve + Kalman/EKF feedback (~5s cadence) │ THIS IS WHERE THE BUG LIVED ▼ LAYER 3 — PREDICTIVE ML-enhanced noise tuning (learns over time)

The Kalman filter was chosen for equatorial mounts because their tracking geometry is linear in RA/Dec — a 4-state model (RA position, RA velocity, Dec position, Dec velocity) captures the essential dynamics. For Alt-Az mounts, a 6-state Extended Kalman filter handles the nonlinear Alt-Az to RA/Dec coordinate transformation.

Bug #1: The Transposed Matrix

K is 4×2. Transposing it yields 2×4. Multiplying by a 2-element innovation gives a 2-vector. The state vector needs 4. Index overflow. Every cycle. For every equatorial mount user. From the audit log

The Kalman gain matrix K is computed as P·HT·S-1, where P is the 4×4 covariance, H is the 2×4 measurement Jacobian, and S is the 2×2 innovation covariance. K is therefore 4×2. The update step should compute state += K · y, where y is the 2-element innovation (RA error, Dec error). The code instead computed Transpose(K) · y, which produces a 2-vector that overflows the 4-element state array.

The tracking loop caught the IndexOutOfRangeException and continued — a design choice intended to keep the mount tracking even if the filter failed. But this meant the failure was invisible. The mount fell back to bare sidereal tracking with no correction, and the user saw no error message.

The Fix

KalmanFilter.Update():
  BEFORE:  state += MatrixVectorMultiply(Transpose(K), y)  // 2×4 × 2 = 2-vector → overflow
  AFTER:   state += MatrixVectorMultiply(K, y)             // 4×2 × 2 = 4-vector → correct

Bug #2: EKF Threw Its Correction Away

The Extended Kalman filter for Alt-Az mounts computed its optimal tracking rate in GetOptimalTrackingRate — then discarded the result. The method calculated azimuth and altitude rates from the EKF's estimated drift bias states, mapped them to RA/Dec via a numerical Jacobian, and returned the correction. But the code then overwrote this with a hardcoded sidereal-plus-proportional fudge. The EKF's math never reached the mount.

This was the most insidious kind of bug: the code did not crash. It produced output. Tests could verify the EKF state evolution. But the actual rate command sent to the mount was always the hardcoded approximation, regardless of what the filter estimated.

The Fix

GetOptimalTrackingRate():
  BEFORE:  compute azRate/altRate from EKF state → discard → return hardcoded fudge
  AFTER:   derive correction from bias states b_az, b_alt → Jacobian map to RA/Dec drift
           → rate = SIDEREAL - drift×gain - error/10×gain (clamped ±30″/s)

Bug #3: MatrixInverse Returns Identity

KalmanFilter.MatrixInverse returned the identity matrix for any input larger than 2×2. No current caller hit this path — the Kalman filter's state is exactly 4×4 and the innovation covariance is exactly 2×2, both within the working range. But any future expansion (adding acceleration states, for instance) would silently corrupt the Kalman gain computation.

Code that computes but never applies its result is the most dangerous kind of bug. It passes tests. It produces output. It does nothing. Audit finding

The fix replaced the stub with a general n×n Gauss-Jordan elimination with partial pivoting — a standard numerical linear algebra routine that handles any matrix size correctly.

Lessons Learned

1. Silent Exception Swallowing Kills Features

The tracking loop caught exceptions to keep the mount tracking during filter failures. This was a reasonable design for transient numerical issues, but it hid a persistent mathematical error for months. Safety-critical code paths should log at ERROR level before continuing, not silently swallow.

2. "Working" Tests ≠ Working Code

131 tests passed because none tested the actual Kalman update math — they tested the surrounding infrastructure, the ASCOM wrappers, the UI state management. Unit tests must exercise the mathematical core, not just the API surface.

3. Dead Code Feels Like Working Code

The EKF's rate was computed, discarded, and replaced with a hardcoded value. It "worked" in the sense that it produced output and passed tests. But the output never reached the mount. Code that computes but never applies its result is worse than code that crashes — it's invisible.

4. Copy-Paste Math Needs Dimensional Analysis

The transposed-multiply was likely a transcription from a textbook where K had different dimensions. Adding runtime assertions that matrix dimensions match — not just compile-time checks — would have caught this at the first plate solve.
