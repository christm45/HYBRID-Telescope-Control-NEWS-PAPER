Building the Voice That Talks to 47 Telescope Tools

From microphone audio to telescope motion in under 3 seconds — the full architecture of a conversational voice pipeline powered by Whisper, Silero VAD, and the Model Context Protocol.

MCP Telescope Tools

Exposed via the Model Context Protocol — the same standard Anthropic's Claude uses to connect to tools. Any MCP-compatible AI client can drive the telescope.

HYBRID Telescope Control is the first amateur astronomy app to integrate conversational voice control not as a novelty feature, but as the primary interaction model. You say "slew to M31, plate solve, and start imaging for 90 minutes" and the scope does it. The voice pipeline is a multi-stage architecture: capture, transcribe, parse, dispatch, respond — and it runs from microphone audio to telescope motion in under three seconds.

The system supports three interchangeable voice backends: Claude Desktop via MCP for cloud AI reasoning, a local llama.cpp server for fully offline operation at remote observatories, and a built-in Whisper grammar parser as a zero-dependency fallback. The user picks the mode that fits their session. The hardware commands are identical regardless of which backend processes the voice input.

The Pipeline: Audio to Motion

The voice pipeline begins at the microphone. Audio is captured using Python's sounddevice library with numpy for buffer management. Silero VAD (Voice Activity Detection) monitors the audio stream and detects when the user is speaking — this replaces wake-word detection, which proved unreliable near cooling fans and wind noise. Push-to-talk is more reliable in an observatory environment.

Speech-to-Text: Faster Whisper

The STT engine uses faster-whisper, a CTranslate2 port of OpenAI's Whisper model, running the small.en variant. It was chosen for its balance of accuracy and speed — the small model transcribes in near-real-time on modern hardware while handling the disfluencies and hesitations of natural speech.

Whisper was trained on general internet text, not astronomical object names. "M31" might come through as "M 31" or "am 31". "Andromeda" might be "and a meda". The correction pipeline maps these errors before the parser sees them. Engineering notes

62 built-in STT corrections handle Whisper's most common astronomical transcription errors. "M31" and "am 31" both map to the canonical "M31". "Pleiadies" maps to "M45". "Orion nebula" maps to "M42". These corrections run before the grammar parser, so the parser only sees clean canonical names.

Command Parsing: 55+ Actions

The telescope parser is a grammar-based system defined in telescope_grammar.py with parser tables generated at build time. It maps natural language phrases to canonical actions — 55 distinct actions covering mount control, camera operations, focuser, tracking, scheduling, weather, session management, and emergency procedures.

Mount Actions

goto_target, goto_coordinates, park, unpark, find_home, stop_move, move_axis, emergency_stop, abort_slew

Camera Actions

quick_capture, start_imaging, stop_imaging, pause_imaging, resume_imaging, autofocus, plate_solve

Tracking Actions

start_tracking, stop_tracking, get_tracking_status, get_kalman_state, get_ml_status, start_advanced_tracking

Scheduler Actions

add_target, get_night_plan, start_scheduler, stop_scheduler, skip_target, get_scheduler_targets

Each action supports 3 to 8 natural language phrase variants. "Slew to M31", "go to M31", "point at Andromeda", "track M31" all map to the same goto_target action. Fuzzy matching handles minor misrecognitions. The parser returns a canonical action name, target type, and parsed parameters — the dispatcher does not need to understand natural language.

MCP: The Model Context Protocol

The Model Context Protocol is an open standard (Anthropic, 2024) that gives AI assistants access to external tools via a structured interface. HYBRID exposes 47 telescope tools through an MCP server built on .NET 8. Claude Desktop, VS Code Copilot, or any MCP-compatible client can discover and call these tools.

Five MCP Servers

  • Telescope — 47 tools for mount, camera, focuser, tracking
  • Playwright — browser automation
  • Filesystem — read/write project files
  • Fetch — HTTP requests
  • Memory — persistent knowledge graph

What makes MCP powerful is tool chaining. A user can say "slew to M81, plate solve, autofocus, and start imaging for 30 minutes with 180-second exposures" — and the AI breaks this into a sequence of MCP tool calls: goto_target, plate_solve, autofocus, start_imaging. Each tool call is dispatched to the ASCOM API. The AI handles the orchestration; the hardware handles the motion.

Why Python + C#?

The voice pipeline runs in Python because the key libraries — faster-whisper, Silero VAD, gTTS — have no C# equivalents with comparable performance. The main application runs in C# because WPF is the native Windows UI framework and ASCOM integration is C#-native. The bridge is a clean subprocess IPC: Python processes audio and produces a command string; C# receives it via stdout and dispatches to ASCOM. No shared state, no DLL conflicts, no runtime version issues.

Design Principles

Three principles guided the voice architecture. First, reliability over intelligence — push-to-talk over wake-word, grammar parsing over free-form NLU, deterministic dispatch over probabilistic reasoning. Second, graceful degradation — if Claude is unavailable, the local LLM takes over; if the LLM is unavailable, the grammar parser handles it; if the grammar fails, the user falls back to clicks. Third, transparent failure — every voice command produces visible feedback in the UI and spoken TTS response, so the user always knows what happened and why.
