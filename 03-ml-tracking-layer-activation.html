Making the ML Layer Actually Work

The SDCA pipeline trained a single-output regressor. Confidence was always zero. The gate never passed. The advertised ML adaptivity contributed nothing — until now.
By Bogdan • June 11, 2026 • Dual-Axis SDCA Regression
0
Confidence (was)

The ML confidence field was always zero. The Confidence > 0.5 gate never passed. The ML noise-tuning path was dead code.

Since v1.2.0, HYBRID Telescope Control advertised "ML-enhanced tracking" — a machine learning layer that adapts the Kalman filter's noise parameters based on observed drift patterns. The feature appeared in the UI, appeared in the documentation, and did absolutely nothing. A deep audit in June 2026 discovered why: the ML pipeline was architecturally complete but mathematically inert.

The SDCA (Stochastic Dual Coordinate Ascent) pipeline in ML.NET trained a single-output regressor with the label RADrift. ML.NET's prediction engine only populated the Score field. Every other field the consumer code read — Confidence, ProcessNoise, MeasurementNoise, DecDrift — was left at its default value of zero. The Confidence > 0.5 gate in the filter's Update method therefore never passed, and the ML noise-tuning path was never executed.

The 3-Layer Tracking Model

To understand what was broken, it helps to understand the full tracking architecture. HYBRID operates three layers simultaneously:

LAYER 1 — PASSIVE Sidereal / ASCOM firmware tracking (always on) LAYER 2 — ACTIVE Plate-solve + Kalman/EKF feedback (~5s cadence) LAYER 3 — PREDICTIVE ML-enhanced noise tuning (learns over time) ← THIS WAS DEAD

Layer 3 was supposed to observe the Kalman filter's prediction errors over time, train a regression model on the 24-feature vector extracted from each plate-solve cycle, and then dynamically adjust the filter's process and measurement noise covariance. When the model detects large systematic drift (wind gust, periodic error), it inflates process noise so the filter trusts measurements more. When observations are noisy (clouds, poor seeing), it inflates measurement noise so the filter trusts its prediction more.

The Root Cause: Single-Output Regression

The SDCA pipeline trained one model with Label=RADrift. ML.NET only populated Score. Confidence, ProcessNoise, MeasurementNoise, DecDrift — all zero. The gate never passed. The advertised ML adaptivity contributed nothing. Audit finding, DEV LOG 012

The problem was a design oversight in the training pipeline. The original implementation trained a single SDCA regressor on the RA drift label. But the consumer code expected five fields: RADrift, DecDrift, Confidence, ProcessNoise, and MeasurementNoise. Only RADrift was populated (via Score). The other four were struct fields with zero defaults.

The Confidence field was meant to represent the model's R² (coefficient of determination) — how well the trained model explains the observed data. With R² stuck at zero, the > 0.5 gate was permanently closed. Even if the noise fields had been populated, they would have been zero — which would have collapsed the covariance matrix to zero, making the filter ignore all measurements.

The Fix: Dual-Axis SDCA

The fix trains two SDCA regressors per filter — one with Label=RADrift, one with Label=DecDrift — sharing the same feature concatenation. Two PredictionEngine instances are created, one per axis. A new BuildPrediction helper assembles a fully populated prediction:

FieldBefore (Broken)After (Fixed)
RADriftScore (partial)RA-model.Score (real prediction)
DecDriftAlways 0Dec-model.Score (real prediction)
ConfidenceAlways 0clamp(R², 0, 1) from trained model
ProcessNoiseAlways 0clamp(1 + |drift|/baseline, 0.1, 10)
MeasurementNoiseAlways 0clamp(1 + scatter/baseline, 0.1, 10)

The confidence gate now has a meaningful value. When the model's R² exceeds 0.5 — meaning the model genuinely explains more than half the variance in the drift data — the ML layer engages. When R² is below 0.5, the filter falls back to the raw Kalman/EKF without ML noise tuning. The gate is no longer decorative.

Noise Scaling: How It Adapts

The noise scaling is the core of the adaptivity. Process noise scale is computed as clamp(1 + |drift|/baseline, 0.1, 10.0). When the ML model detects large drift, the scale increases, telling the filter to trust measurements more. Measurement noise scale is clamp(1 + scatter/baseline, 0.1, 10.0). When observations are noisy, the scale increases, telling the filter to trust its prediction more.

When the ML model detects large drift (wind gust, periodic error spike), it inflates the process noise so the filter trusts measurements more. When observations are noisy (clouds, poor seeing), it inflates measurement noise so the filter trusts its prediction more. This is the adaptivity that was advertised but never delivered. Engineering notes

Both scales are hard-clamped to [0.1, 10.0]. This prevents covariance collapse (scale approaching zero, filter ignores everything) and covariance explosion (scale approaching infinity, filter ignores nothing). The clamp is a safety boundary that guarantees numerical stability regardless of what the ML model predicts.

The 24-Feature Vector

The ML model sees a 24-feature vector at each training step, extracted from the plate-solve history:

Feature Categories

Position: RA/Dec errors, rates, hour angle, altitude, azimuth

Temporal: Timestamp, UT hour, LST hour, Julian date fraction

Environmental: Temperature, humidity, wind, pressure

Statistical: Error std dev, drift trend, innovation magnitude

These features capture the full context of each measurement: where the mount is pointing, what time it is, what the weather is doing, and how the filter has been performing. The SDCA regressor learns which features correlate with drift, and uses that correlation to predict future drift from the current feature vector.

Thread Safety and Persistence

The fix also addressed two operational concerns. First, thread safety: the WebServer thread calls PredictDriftML while the UI thread can trigger ForceRetrain, and the EKF retrains on a background Task. A new modelLock guards engine swaps and paired Predict calls to prevent torn reads. Second, persistence: both models are saved side by side (kalman_ml_model.zip + _dec.zip for KF; ekf_ml_model.zip + _dec.zip for EKF). A freshly loaded model reads the persisted R² from metrics.json so the confidence gate works without retraining.

The ML-enhanced tracking layer is now active. On the first session, it collects training data, trains both RA and Dec models, evaluates R², and engages if the threshold is met. On subsequent sessions, saved models load immediately and the confidence gate opens from the first plate solve. The system adapts to the mount's actual behavior in real time — which is what was promised eighteen months ago.
