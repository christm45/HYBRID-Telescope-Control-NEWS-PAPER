Activating the ML Tracking Layer: Dual-Axis SDCA Regression

The SDCA training pipeline was architected for single-axis regression. Extending it to dual-axis prediction — with meaningful confidence scoring and adaptive noise scaling — activates the ML layer's full contribution to tracking precision.
By Bogdan • June 11, 2026 • Dual-Axis SDCA Regression
R²
Confidence gate activated

The ML confidence field now carries a meaningful R² value from the trained model. The Confidence > 0.5 gate engages when the model explains sufficient variance in the drift data.

Since v1.2.0, HYBRID Telescope Control has included an ML-enhanced tracking layer — a machine learning component designed to adapt the Kalman filter's noise parameters based on observed drift patterns. The architectural framework was complete: the 24-feature extraction pipeline, the SDCA regression engine, the confidence gate, and the noise scaling formulas were all in place. A comprehensive review in June 2026 identified that the training pipeline needed extension from single-axis to dual-axis regression to fully activate the ML layer's intended contribution to tracking precision.

The SDCA (Stochastic Dual Coordinate Ascent) pipeline in ML.NET originally trained a single-output regressor with the label RADrift. ML.NET's prediction engine populated the Score field with the RA drift prediction. However, the consumer code expected five fields: RADrift, DecDrift, Confidence, ProcessNoise, and MeasurementNoise. The single-axis approach left four fields at their default values, which meant the confidence gate could not evaluate model quality and the noise scaling formulas had no input data to work with.

The 3-Layer Tracking Model

HYBRID operates three tracking layers simultaneously. Understanding the full architecture clarifies the ML layer's role within the system:

LAYER 1 — PASSIVE Sidereal / ASCOM firmware tracking (always on) LAYER 2 — ACTIVE Plate-solve + Kalman/EKF feedback (~5s cadence) LAYER 3 — PREDICTIVE ML-enhanced noise tuning (learns over time) ← ACTIVATED in this release

Layer 3 observes the Kalman filter's prediction errors over time, trains a regression model on the 24-feature vector extracted from each plate-solve cycle, and dynamically adjusts the filter's process and measurement noise covariance. When the model detects large systematic drift (wind gust, periodic error), it inflates process noise so the filter trusts measurements more. When observations are noisy (clouds, poor seeing), it inflates measurement noise so the filter trusts its prediction more.

The Training Pipeline Extension

The SDCA pipeline trained one model with Label=RADrift. ML.NET populated Score. The remaining fields — Confidence, ProcessNoise, MeasurementNoise, DecDrift — retained their default values. Extending to dual-axis regression activates the complete prediction pathway. Engineering analysis, DEV LOG 012

The original implementation trained a single SDCA regressor on the RA drift label. The consumer code expected five fields, but only one was populated via the Score output. The other four were struct fields with zero defaults. The Confidence field, designed to represent the model's R² (coefficient of determination), remained at zero — which meant the > 0.5 gate was permanently closed and the ML noise-tuning pathway was not engaged.

The extension trains two SDCA regressors per filter — one with Label=RADrift, one with Label=DecDrift — sharing the same feature concatenation. Two PredictionEngine instances are created, one per axis. A new BuildPrediction helper assembles a fully populated prediction:

RADriftScore (partial)RA-model.Score (real prediction)
DecDriftDefault (0)Dec-model.Score (real prediction)
ConfidenceDefault (0)clamp(R², 0, 1) from trained model
ProcessNoiseDefault (0)clamp(1 + |drift|/baseline, 0.1, 10)
MeasurementNoiseDefault (0)clamp(1 + scatter/baseline, 0.1, 10)

The confidence gate now receives a meaningful R² value. When the model's R² exceeds 0.5 — meaning the model explains more than half the variance in the drift data — the ML layer engages. When R² is below 0.5, the filter operates with baseline Kalman/EKF parameters. The gate provides a quality threshold that ensures the ML layer only contributes when its predictions are statistically meaningful.

Noise Scaling: The Adaptivity Engine

The noise scaling formulas translate ML predictions into Kalman filter tuning adjustments. Process noise scale is computed as clamp(1 + |drift|/baseline, 0.1, 10.0). When the ML model detects large drift, the scale increases, telling the filter to trust measurements more. Measurement noise scale is clamp(1 + scatter/baseline, 0.1, 10.0). When observations are noisy, the scale increases, telling the filter to trust its prediction more.

When the ML model detects large drift — a wind gust or periodic error spike — it inflates the process noise so the filter trusts measurements more aggressively. When observations are noisy — clouds or poor seeing — it inflates measurement noise so the filter relies more on its internal prediction. This is the adaptive capability that the architecture was designed to deliver. Engineering notes

Both scales are hard-clamped to [0.1, 10.0]. This prevents covariance collapse (scale approaching zero, filter ignores everything) and covariance explosion (scale approaching infinity, filter ignores nothing). The clamp is a safety boundary that guarantees numerical stability regardless of what the ML model predicts.

The 24-Feature Vector

The ML model processes a 24-feature vector at each training step, extracted from the plate-solve history:

Feature Categories

Position: RA/Dec errors, rates, hour angle, altitude, azimuth

Temporal: Timestamp, UT hour, LST hour, Julian date fraction

Environmental: Temperature, humidity, wind, pressure

Statistical: Error std dev, drift trend, innovation magnitude

These features capture the full context of each measurement: where the mount is pointing, what time it is, what the weather is doing, and how the filter has been performing. The SDCA regressor learns which features correlate with drift, and uses that correlation to predict future drift from the current feature vector.

Thread Safety and Model Persistence

The extension also addressed two operational concerns. First, thread safety: the WebServer thread calls PredictDriftML while the UI thread can trigger ForceRetrain, and the EKF retrains on a background Task. A modelLock guards engine swaps and paired Predict calls to prevent torn reads. Second, persistence: both models are saved side by side (kalman_ml_model.zip + _dec.zip for KF; ekf_ml_model.zip + _dec.zip for EKF). A freshly loaded model reads the persisted R² from metrics.json so the confidence gate works from the first plate solve without retraining.

⚙ Engineering Improvements

The dual-axis extension represents a structural improvement to the ML pipeline that enhances both correctness and maintainability.

Complete Prediction Pathway

The training pipeline now populates all five prediction fields that the consumer code expects. This eliminates the gap between what the architecture intended and what the implementation delivered, ensuring the ML layer's full capabilities are available to the tracking system.

Independent Axis Models

Training separate models for RA and Dec drift reflects the physical reality that these error sources have different characteristics. RA error is driven primarily by periodic error in the worm gear and atmospheric refraction modeling. Dec error is driven by polar alignment error, flexure, and wind loading. Independent models capture these distinct dynamics more accurately than a single combined model.

Thread-Safe Model Lifecycle

The modelLock mutex ensures that model swaps — whether triggered by retraining, session load, or API request — do not produce torn reads across threads. This is particularly important for the WebServer thread, which calls PredictDriftML on every plate-solve cycle while the UI or background tasks may trigger retraining simultaneously.

📈 Reliability Evolution

The activation of the ML layer enhances tracking reliability through adaptive noise tuning that responds to real-time conditions.

Adaptive Noise Parameters

With the confidence gate operational, the Kalman filter's noise covariance matrices now adapt to current conditions. On steady nights with good seeing, the ML model's R² exceeds the threshold and the filter operates with optimized noise parameters. On turbulent nights, the model's R² falls below the threshold and the filter uses baseline parameters — a graceful fallback that prevents degradation.

Model Persistence Across Sessions

Saved models load at session start, providing ML augmentation from the first plate solve. The persisted R² value ensures the confidence gate evaluates the loaded model's quality before engaging. This means a model trained on similar conditions (same mount, same temperature range) can contribute immediately, while a model from significantly different conditions will not engage until fresh training data validates it.

Backward Compatibility

Legacy single-axis models import safely, with the RA model reused for Dec prediction as a fallback. This ensures that existing installations upgrade seamlessly without losing their trained models.

Lessons Learned

1. Architecture vs. Implementation

The ML layer's architecture was sound — the 24-feature vector, the confidence gate, the noise scaling formulas were all well-designed. The gap was in the training pipeline's implementation, which trained only one axis. This distinction matters: the design was validated by the fact that the fix was straightforward once the gap was identified.

2. Default Values as Silent Failures

Struct fields defaulting to zero produced no errors, no exceptions, no test failures. The system ran correctly — it just didn't run the ML pathway. This pattern — where default values satisfy compilation and basic testing but prevent intended functionality — is worth watching for in any pipeline with optional components.

3. Confidence as a Quality Gate

The R² confidence gate proved its design value. Rather than always engaging the ML layer (which could degrade tracking if the model is poorly trained), the gate ensures the model must demonstrate statistical quality before influencing the filter. This principle — requiring proof of quality before engagement — applies broadly to adaptive systems.
