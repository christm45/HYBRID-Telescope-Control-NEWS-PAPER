When Your Telescope Is One ngrok URL Away From Anyone

A 4-area deep audit found that AuthEnabled defaulted to false — anyone with the tunnel URL could issue mount commands, change tracking, or shut down the observatory remotely.

Vulnerabilities found

Two critical, three high, one medium — across voice, camera, mount, and web subsystems. All fixed in a single audit campaign.

On June 10, 2026, a comprehensive security audit of HYBRID Telescope Control's four major subsystems — voice, camera/imaging, mount/ASCOM/scheduler, and web/MCP/cloud — uncovered six real vulnerabilities. Two were critical: one could expose the entire observatory to remote attack via the internet, the other could physically damage equipment by driving the telescope into its own pier.

The audit was systematic. Each area was reviewed line by line against the source code, with every finding verified as a real bug (not a theoretical risk) before being classified. The result was nine build-verified commits that addressed every critical and high-severity finding in a single campaign.

Critical — Web Authentication Disabled by Default

AuthEnabled defaulted to false and was never set true on tunnel start. Starting an ngrok tunnel exposed /api/command/* and /mcp/* to the internet with no credentials. Audit finding

The embedded web server's AuthEnabled flag defaulted to false. This was a convenience default — during local development, authentication is unnecessary friction. But when the user started an ngrok tunnel for remote access, the flag was never set to true. The result: anyone with the ngrok URL could access /api/command/* and /mcp/* endpoints without any credentials.

The problem compounded with CORS. The ngrok-to-localhost proxy forwarded requests with standard HTTP headers, but the server's IsLocalRequest check looked at the connection's origin — which appeared local because the traffic came through the tunnel's localhost endpoint. CORS was set to * for local requests, meaning any website could issue cross-origin commands to the mount.

The Fix

SetAuthEnabled is now forced ON at both tunnel-start paths (and OFF on stop). The auth gate is scoped to non-local /api/ and /ws/ endpoints. IsLocalRequest now treats X-Forwarded-* headers as WAN traffic. The /mcp/* endpoint requires non-local origin plus Bearer token or API key.

Critical — The Meridian Flip That Could Hit the Pier

The scheduler's meridian flip was supposed to rotate the mount 180° as the target crossed the meridian, keeping the telescope on the same side of the pier. The implementation called a same-target re-slew — a hopeful maneuver that re-pointed at the same coordinates without verifying the pier side changed. If the flip failed silently, the imaging sequence would continue on the wrong side of the pier, with the OTA approaching physical collision.

Before (Broken)

Scheduler calls re-slew to same target. Does not wait for completion. Does not verify pier side. Returns Success on failure.

After (Fixed)

Scheduler calls real ExecuteMeridianFlipAsync. Waits for slew. Verifies pier side changed. Halts + stops session on failure.

SafeParkAsync was also hardened: it now calls Abort + StopTracking before Park, with a StopAllAxes fallback if Park fails. The flip no longer guesses pier side when the ASCOM driver returns "Unknown".

High — Process Leaks and Session Corruption

Two high-severity issues affected reliability. Voice bridge Python processes and the local llama.cpp server were not stopped on app close — they leaked as zombie processes, with the llama server consuming significant GPU memory. StopLocalLlmServer via taskkill /F /T /PID now runs on app close.

Session state was also vulnerable. session_state.json — containing open tabs, todo progress, and conversation history — was written directly without atomicity. A crash mid-write could corrupt the file, losing the entire session. The fix implements lock + atomic temp-then-move: write to a .tmp file, rename on success. No partial writes ever reach the real file.

High — FITS Reader Corrupted Autofocus

The FITS reader used by the autofocus pipeline did not handle multi-block headers, BZERO/BSCALE scaling, or all BITPIX values. Full-header and float frames produced corrupted FWHM/HFR measurements, leading to bad autofocus curves. The fix delegated FITS reading to ImageLoader, which handles all FITS variants correctly.

Medium — Process I/O and UI Freeze

Start-tunnel froze the UI for up to 10 seconds because tunnelStartTask.GetAwaiter().GetResult() blocked the dispatcher. The fix uses fire-and-forget Task.Run with async AddMessage on completion, returning "starting..." immediately. The zenith azimuth rate was also capped at 5 deg/s — it previously returned ~4 deg/s at the zenith singularity, which could command a runaway slew.
