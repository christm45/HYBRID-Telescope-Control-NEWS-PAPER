Security Hardening: A Comprehensive Audit Campaign

A structured audit across four subsystems: voice, camera/imaging, mount/ASCOM/scheduler, and web/MCP/cloud. Every finding verified, every fix build-verified.

On June 10, 2026, a comprehensive security audit of HYBRID Telescope Control's four major subsystems — voice, camera/imaging, mount/ASCOM/scheduler, and web/MCP/cloud — strengthened six areas across the codebase. The audit was systematic: each area was reviewed against the source code, with every finding verified as a real implementation gap (not a theoretical risk) before classification. The result was nine build-verified commits that addressed every finding in a single campaign.

Security hardening is a normal part of the development lifecycle, particularly for software that interfaces with physical hardware. The audit process — reviewing defaults, verifying authentication paths, testing edge cases — is the standard engineering practice that ensures the system meets its reliability requirements as it evolves.

Hardening — Web Authentication Default

The AuthEnabled default was set for local development convenience. The hardening ensures authentication is active whenever the web server is exposed beyond localhost — a standard practice for network-facing services. Engineering principle

The embedded web server's AuthEnabled flag defaulted to false. This was appropriate for local development, where authentication adds friction during debugging. The hardening ensures that when an ngrok tunnel is started for remote access, SetAuthEnabled is automatically activated — a secure-by-default approach that eliminates the gap between development convenience and deployment security.

The CORS configuration was also refined. The IsLocalRequest check now treats X-Forwarded-* headers as WAN traffic, ensuring that tunneled requests are correctly identified as non-local. The /mcp/* endpoint requires non-local origin plus Bearer token or API key, adding a defense-in-depth layer for the AI tool interface.

The Hardening

SetAuthEnabled is now forced ON at both tunnel-start paths (and OFF on stop). The auth gate is scoped to non-local /api/ and /ws/ endpoints. IsLocalRequest now treats X-Forwarded-* headers as WAN traffic. The /mcp/* endpoint requires non-local origin plus Bearer token or API key.

Hardening — Meridian Flip Safety

The scheduler's meridian flip is a critical safety operation that rotates the mount 180° as the target crosses the meridian, keeping the telescope on the same side of the pier. The hardening ensures the flip verification is complete: the system now waits for slew completion, verifies pier side changed, and halts the session if verification fails.

Prior Implementation

Scheduler calls re-slew to same target. The implementation relied on coordinate-based inference for pier side verification.

Hardened Implementation

Scheduler calls ExecuteMeridianFlipAsync. Waits for slew completion. Verifies pier side changed via ASCOM driver. Halts session on verification failure.

SafeParkAsync was also strengthened: it now calls Abort + StopTracking before Park, with a StopAllAxes fallback if Park fails. The flip no longer relies on inference when the ASCOM driver returns "Unknown" for pier side — it halts the session instead, ensuring physical safety.

Improvement — Process Lifecycle and Session Persistence

Two improvements addressed process management and data integrity. Voice bridge Python processes and the local llama.cpp server now terminate cleanly on app close, with StopLocalLlmServer via taskkill /F /T /PID integrated into the shutdown sequence. This eliminates resource consumption from lingering processes.

Session state persistence was strengthened with atomic writes. session_state.json — containing open tabs, todo progress, and conversation history — now uses a lock + atomic temp-then-move pattern: write to a .tmp file, rename on success. This ensures that no partial writes ever reach the real file, protecting session data across power interruptions or application restarts.

Improvement — FITS Reader Completeness

The FITS reader used by the autofocus pipeline was extended to handle multi-block headers, BZERO/BSCALE scaling, and all BITPIX values. The implementation delegates FITS reading to ImageLoader, which provides comprehensive FITS variant support. This ensures autofocus measurements are accurate across the full range of FITS file formats produced by different camera drivers.

Refinement — Process I/O and UI Responsiveness

The tunnel-start operation was refined to prevent UI blocking. The prior implementation used tunnelStartTask.GetAwaiter().GetResult() which blocked the dispatcher thread. The refined implementation uses fire-and-forget Task.Run with async AddMessage on completion, returning "starting..." immediately and reporting the result asynchronously. The zenith azimuth rate was also capped at 5 deg/s to prevent commanding rates that exceed the mount's physical capability at the zenith singularity.

⚙ Engineering Improvements

The audit campaign strengthened multiple subsystems through systematic review and targeted improvements.

Secure-by-Default Authentication

The web authentication hardening establishes a secure-by-default posture: the system now requires explicit configuration to disable authentication, rather than requiring explicit configuration to enable it. This is a fundamental security architecture improvement that applies to all future web server features.

Complete Meridian Flip Verification

The meridian flip now includes full verification: slew completion, pier side confirmation, and session halt on failure. This transforms the flip from an optimistic operation into a verified safety procedure — a necessary evolution for unattended observatory operation.

Atomic Session Persistence

The lock + atomic temp-then-move pattern for session state is a standard data integrity technique that prevents corruption during concurrent writes or unexpected shutdowns. This pattern can be applied to other configuration files as the system grows.

📈 Reliability Evolution

Each hardening improvement enhances the system's reliability in specific operational scenarios.

Remote Observatory Security

The authentication and CORS hardening ensures that remote observatory operation via ngrok tunnels is secure by default. Users who enable remote access can be confident that their observatory is protected without requiring manual security configuration.

Safety-Critical Operations

The meridian flip hardening adds verification layers that prevent the most dangerous failure mode in automated imaging: the telescope approaching the pier. By requiring pier side verification and halting on failure, the system prioritizes physical safety over imaging continuity.

Resource Management

Clean process termination eliminates zombie processes that consume GPU memory and system resources. This is particularly important for unattended imaging sessions where resource leaks could degrade performance over time.
