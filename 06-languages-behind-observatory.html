Languages & Architecture

The Languages Behind the Observatory

Five programming languages, two runtime environments, one telescope control system. A map of where every line of code lives and why it lives there.
By Bogdan • Architecture Deep Dive

Programming Languages

C#, Python, C++, HTML/CSS/JS, PowerShell/Batch/NSIS — each chosen for what it does best.

HYBRID Telescope Control is not a single-language project. It is a polyglot system where five programming languages each handle the subsystem they are best suited for. C# runs the application and hardware integration. Python handles voice recognition and image processing scripts. C++ provides native math performance. HTML, CSS, and JavaScript build the remote-access web dashboard. PowerShell, Batch, and NSIS handle build, packaging, and deployment.

This is not accidental complexity. Each language boundary maps to a genuine engineering constraint: WPF requires C#, faster-whisper requires Python, ASCOM integration requires .NET, the web dashboard requires browser-native technologies, and the installer requires Windows-native tooling. The result is a system where the language choices are invisible to the end user — they see one application, one installer, one experience.

C# — The Backbone

C# at a Glance

200+ source files

.NET Framework 4.8 (main app)

.NET 8 (MCP server)

C# 7.3 language version

WPF for UI

C# is the primary language, comprising roughly 80% of the codebase. The main application — every UI tab, every ASCOM wrapper, every tracking algorithm, every image processing pipeline, the web server, the scheduler, the session manager — is written in C# targeting .NET Framework 4.8 with WPF for the user interface.

The choice of C# was driven by three constraints. First, ASCOM Alpaca is a .NET-native protocol — the HTTP REST API for telescope control was designed for C# consumers. Second, WPF provides the rich data-binding, template system, and dark-theme capabilities needed for an observatory application. Third, .NET Framework 4.8 is pre-installed on every Windows 10/11 machine, eliminating runtime dependencies for end users.

The MCP server runs on .NET 8 — a separate project, compiled independently, because the server needs modern .NET features (minimal APIs, source generators) that .NET Framework 4.8 does not support. The two .NET runtimes coexist cleanly: the main app spawns the MCP server as a child process and communicates via HTTP.

Python — The Voice and the Post-Processor

Python at a Glance

Python 3.11 (bundled)

faster-whisper STT

Silero VAD

gTTS / edge-tts

numpy post-processing

Python handles two critical subsystems that have no viable C# alternatives: voice recognition and certain image processing scripts. The voice pipeline uses faster-whisper (SYSTRAN's CTranslate2 port of OpenAI's Whisper) for speech-to-text, Silero VAD for voice activity detection, and gTTS or edge-tts for text-to-speech feedback. These libraries are Python-native — they depend on PyTorch and CTranslate2, neither of which has a production-quality C# binding.

The post-processing scripts use numpy for AI denoising (a-trous wavelet transforms), star removal (morphological opening), and super-resolution (bicubic upscale + unsharp mask). These were reimplemented from scratch in numpy because the original Seti Astro Suite Pro scripts were reference implementations, not redistributable code. The numpy implementations are dependency-light — they run on any machine with Python and numpy, no GPU required.

Python is bundled in the installer as an embeddable CPython 3.11 distribution (~1.4 GB) with all pip packages pre-installed. The main C# application launches Python scripts as subprocesses, communicates via stdout/stdin JSON, and manages their lifecycle. This clean process boundary means Python version conflicts, DLL hell, and runtime incompatibilities are structurally impossible.

C++ — The Native Math Layer

The C++ layer is small but performance-critical. Two projects — HybridTelescopeControl.Core and HybridTelescopeControl.Shared — provide native astro math utilities called via P/Invoke from C#. These include coordinate transformations, matrix operations, and numerical routines that benefit from native compilation.

The C++ layer also wraps the Canon EDSDK for direct live view integration with Canon DSLRs — a P/Invoke bridge to Canon's proprietary native DLL. This bypasses the ASCOM still-exposure workaround and provides real-time electronic viewfinder on supported bodies.

HTML/CSS/JavaScript — The Remote Observatory

Web Dashboard

7 tabs: Dashboard, Image Preview, Remote Control, Tracking, Mount Model, Scheduler, Logs

1,200+ lines of hand-crafted HTML/CSS/JS

PWA: installable on phone/tablet home screen

Night mode: red-filtered for dark adaptation

The embedded web dashboard is a single-file HTML application served by the C# web server on port 8090. It uses no framework — vanilla JavaScript with CSS Grid and WebSocket for real-time updates. The choice was deliberate: a framework would add build complexity, bundle size, and dependency risk for a dashboard that needs to work reliably in the field on any device.

The dashboard includes a live status panel, image preview with zoom/pan/histogram, remote joystick control for the mount, filter wheel and cooler controls, real-time Kalman filter charts (RA/Dec error, confidence, covariance trace), a scheduler with mosaic wizard, and a live log viewer. It supports PWA installation on phone and tablet home screens, and includes a night-vision mode that turns the UI red to preserve dark adaptation.

PowerShell / Batch / NSIS — Build and Deploy

The build and packaging pipeline uses three scripting languages. BUILD.bat orchestrates MSBuild compilation. BUILD_INSTALLER_NSIS.bat runs the NSIS compiler, stages ~4.9 GB of dependencies, compresses with 7z LZMA2, and creates a self-extracting installer. PowerShell scripts handle deployment via SCP/SSH to the production server, installer upload with progress tracking, and license key generation.

Why This Architecture Works

Clean Boundaries

Every language boundary is a process boundary or a network boundary. No in-process DLL conflicts. No runtime version collisions. Python exceptions don't propagate to the C# app.

Bundled Runtimes

Python is embedded in the installer. The user installs one EXE and gets everything. No "install Python 3.11, then pip install these 20 packages" instructions.

Right Tool, Right Job

C# for WPF and ASCOM. Python for Whisper and numpy. C++ for native math. HTML for the web dashboard. Each language does what it does best.

Testable in Isolation

347+ unit tests run without hardware. The Kalman filter can be tested without a mount. The voice parser can be tested without a microphone. The web dashboard can be tested without a telescope.
