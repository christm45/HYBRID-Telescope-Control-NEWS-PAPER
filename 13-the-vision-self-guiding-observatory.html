The Vision: Building a Self-Guiding Observatory

Why a solo developer in France spent 18 months building the world's first voice-controlled telescope system with predictive AI tracking — and what it means for amateur astronomy.
By Bogdan • June 2026 • Founder's Statement
2024→2026
The Journey

From a frozen fingers problem to a production-hardened observatory system. 18 months. 200+ files. One vision: let the telescope think for itself.

There is a moment every amateur astronomer knows. It is -5°C, the dew is forming on the corrector plate, and you need to change the exposure time. You pull off your glove, find the tiny menu item, click it, type a number, click OK, and put your glove back on. By the time your fingers stop hurting, the seeing has changed and you need to do it all over again. This is how telescope control has worked for decades. This is the problem that started everything.

HYBRID Telescope Control began as a simple question: what if the telescope could talk back? Not in the trivial sense of a wake-word assistant that responds to "Hey Telescope" — that is a toy that can handle "go to M31" but nothing more. The real question was: what if you could say "slew to M31, plate solve, center it, autofocus, and start imaging for two hours with 180-second exposures" — and the telescope just did it? What if the telescope could reason about the sequence, handle errors, and adapt to conditions without human intervention?

The goal was never to build another telescope control application. NINA, SGP, and APT are mature and excellent. The goal was to build a self-guiding observatory — a system where the telescope understands its own state, predicts its own errors, and corrects them before the human even notices. Design philosophy

The Problem With Reactive Systems

Every existing telescope control system is fundamentally reactive. You tell it where to point, and it points. You tell it to take a picture, and it takes a picture. If the mount drifts, you notice and correct. If the focus shifts, you notice and refocus. The human is always in the loop, always watching, always correcting. This works fine for visual observation, but for long-exposure astrophotography — where a single exposure might run for 10 minutes and a single imaging session might run for 8 hours — the reactive model breaks down.

Consider the problem of tracking accuracy. A telescope mount, even a good equatorial mount, drifts. Periodic error from the worm gear, flexure from temperature changes, atmospheric refraction, wind loading, polar alignment error — all of these contribute to drift that degrades image quality. The traditional solution is autoguiding: a second camera watches a guide star and sends correction pulses to the mount. This works, but it is reactive — the guide star must drift before the correction is applied, and the correction is always one cycle behind.

HYBRID takes a different approach: predictive tracking. Instead of waiting for the mount to drift and then correcting, HYBRID predicts the drift before it happens and applies corrections preemptively. The Kalman filter — the same algorithm that guides spacecraft and navigates submarines — estimates the mount's drift state from plate-solve measurements and projects it forward in time. The correction is applied before the drift occurs, not after.

Traditional Autoguiding (Reactive)

Guide star drifts → Camera detects error → Software computes correction → Pulse sent to mount → Mount corrects. Latency: 1-3 seconds. Error accumulates between corrections.

HYBRID Predictive Tracking

Plate solve measures position → Kalman filter predicts drift → ML layer adapts noise parameters → Correction applied preemptively → Mount stays on target. Latency: near-zero. Error prevented before it occurs.

The Three-Layer Architecture

The breakthrough insight was that no single tracking algorithm is sufficient. Atmospheric conditions change, mechanical properties shift with temperature, and different mount geometries require different mathematical models. HYBRID's solution is a three-layer architecture that combines passive, active, and predictive tracking:

LAYER 1 — PASSIVE Sidereal tracking (firmware-level, always on) │ The mount's motors compensate for Earth's rotation at the native sidereal rate. │ No external software needed. This is the baseline. ▼ LAYER 2 — ACTIVE Kalman/EKF feedback (plate-solve every ~5 seconds) │ A plate-solve captures the actual RA/Dec position. The Kalman filter │ computes the error between predicted and actual position. A correction │ rate is applied as a MoveAxis delta on top of sidereal tracking. │ EQUATORIAL: 4-state linear KF (RA pos, RA vel, Dec pos, Dec vel) │ ALT-AZ: 6-state EKF (nonlinear coordinate transform, Jacobian) ▼ LAYER 3 — PREDICTIVE ML-enhanced noise tuning (learns over time) The machine learning layer observes the Kalman filter's prediction errors, trains a regression model on 24 features (position, time, weather, stats), and dynamically adjusts the filter's noise covariance matrices. When the model detects wind gusts → inflates process noise → trusts measurements more When observations are noisy → inflates measurement noise → trusts prediction more

This architecture was not designed in a vacuum. It was shaped by 18 months of real-world use, where each layer was validated, refined, and hardened against the actual conditions of amateur observatory operation. The Kalman filter's matrix dimensions were corrected. The EKF's correction pathway was completed. The ML layer's training pipeline was extended from single-axis to dual-axis regression. Each refinement was driven by a specific failure or limitation encountered in the field.

ASCOM: The Bridge to Hardware

Hardware Compatibility

  • ASCOM Alpaca — Network-based telescope control (REST API)
  • LX200 Serial — Legacy protocol for OnStep and similar controllers
  • Canon EDSDK — Direct live view via P/Invoke
  • UVC — USB webcam support for guide cameras
  • OnStep — Full integration with OnStep controllers

HYBRID does not reinvent hardware communication. It builds on the ASCOM Platform — the industry standard for amateur telescope control in Windows. ASCOM provides a unified API for mount control, camera operations, focuser positioning, filter wheel selection, and observatory automation. HYBRID's ASCOM controller wrapper (1,080 lines of C#) handles GOTO, slew, park, tracking, and pulse guide operations with full error handling and state management.

But ASCOM alone is not enough. The ASCOM standard defines how software talks to hardware — it does not define how software talks to AI. This is where the Model Context Protocol comes in. HYBRID exposes 47 telescope tools through an MCP server, making them available to any MCP-compatible AI assistant. Claude Desktop can discover these tools, understand their parameters, and chain them into complex sequences. The result is a system where the AI handles the orchestration while ASCOM handles the hardware.

YOU (voice or text) │ ▼ AI ASSISTANT (Claude Desktop / Local LLM / Whisper Grammar) │ "Slew to M31, plate solve, autofocus, image 30 min" │ ▼ MCP SERVER (.NET 8, 47 tools) │ goto_target → plate_solve → autofocus → start_imaging │ ▼ ASCOM API (Alpaca REST / LX200 Serial) │ ▼ HARDWARE (Mount + Camera + Focuser + Filter Wheel)

Why Voice Control Matters

Voice control is not a gimmick. It is a genuine usability improvement for the observing workflow. Consider the practical scenarios:

Cold-weather operation. At -5°C, removing gloves to operate a laptop is painful and dangerous — frostbite can develop in minutes. Voice control lets you change targets, adjust exposures, and start imaging sequences without ever touching the keyboard.

Multi-tasking. While the telescope is imaging one target, you can verbally queue the next target, check the weather forecast, or adjust the scheduler — all without interrupting the current imaging session.

Accessibility. Observers with mobility limitations gain a more natural interaction model. Public outreach events become more engaging when you can demonstrate telescope control by speaking rather than clicking.

Complex sequences. The real power is in chaining commands. "Slew to M81, plate solve, autofocus, and start imaging for 30 minutes with 180-second exposures" — this is a single utterance that triggers four MCP tool calls, each dispatched to the ASCOM API. The AI handles the sequence logic; the hardware handles the motion.

Push-to-talk is more reliable near cooling fans and wind. Wake-word false positives are unacceptable when controlling a $5,000 mount. User-initiated voice control is safer than always-on listening. Reliability over intelligence. Design decision log

The Self-Guiding Observatory

The ultimate vision is not just voice control — it is autonomous operation. A self-guiding observatory that can:

Think

The Kalman filter predicts mount drift. The ML layer adapts to changing conditions. The scheduler optimizes target selection based on altitude, weather, and session history. The system reasons about its own state and makes decisions without human input.

Act

ASCOM commands translate predictions into motion. The mount slews to targets. The camera captures exposures. The focuser maintains focus as temperature drops. The filter wheel selects the right filter for each target. The meridian flip executes safely with verification.

Adapt

When clouds roll in, the scheduler pauses and waits. When the seeing improves, the ML layer tightens the noise parameters. When a target sets below the horizon, the system moves to the next target. When the session ends, the telescope parks safely and the system shuts down.

This is not science fiction. Every component exists today. The Kalman filter works. The ML layer adapts. The scheduler sequences. The voice pipeline dispatches. The ASCOM API commands. The remaining challenge is integration — making all these components work together reliably in the harsh conditions of a real observatory, where temperatures drop, clouds appear, and equipment must be trusted to operate unattended through the night.

A Note on Open Source

HYBRID Telescope Control is MIT-licensed and open to contribution. The source code is available for inspection, modification, and redistribution. This is deliberate: astronomical software benefits from transparency. When you are trusting software to control a $5,000 mount through the night, you want to be able to read the code that makes the decisions.

The project has been shaped by 20+ open-source projects: NINA's sequencer architecture, PHD2's guiding algorithms, Stellarium's sky chart rendering, Siril's image stacking, the OnStep community's hardware interface standards, and the MCP ecosystem's tool protocol. HYBRID stands on these shoulders, and the MIT license ensures it gives back.

18 months, one developer, 200+ source files. The audit campaign refined the system for production. The vision is not to replace NINA or SGP — it is to explore a fundamentally different interaction model where the telescope is an active participant in the observing session, not a passive instrument waiting for commands.
