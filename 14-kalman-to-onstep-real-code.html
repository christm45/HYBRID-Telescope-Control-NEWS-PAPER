Kalman to OnStep: The Complete Communication Flow

Tracing every byte from the Kalman filter's mathematical prediction to OnStep's tracking motor — a step-by-step examination of the five-stage pipeline that delivers sub-arcsecond precision.
By Bogdan • July 4, 2026 • Branch: feature/kalman-onstep-pipeline
50–100ms
End-to-end command latency

From Python decision to OnStep motor adjustment — the complete Kalman-to-hardware pipeline executes in under 100 milliseconds per correction.

Every five to fifteen seconds, the HYBRID Telescope Control system performs a remarkable sequence of events: a camera captures a star field, a plate solver determines the exact pointing position, a Kalman filter predicts future drift, Python converts those predictions into OnStep commands, ASCOM Alpaca transmits them over HTTP, and the mount firmware adjusts its tracking rate — all before the next image exposure begins. This article traces the complete communication path with real code.

Kalman = Brain (calculations)  |  ASCOM = Mouth (communication)  |  OnStep = Muscles (action) Core architecture principle

The critical insight that emerges from reading the actual code is that the Kalman filter never communicates directly with OnStep. It performs pure mathematical computation in Python memory — updating state vectors, predicting future drift, and computing optimal tracking rates. A separate layer of Python code then translates those predictions into two distinct types of commands that are transmitted via ASCOM Alpaca's HTTP REST interface. Understanding this separation is essential to grasping why the system achieves ±0.5 arcsecond precision.

The Complete Five-Step Cycle

Every tracking cycle follows the same five-stage pipeline. The stages execute sequentially within a single loop iteration, and the entire sequence repeats every 5 to 15 seconds depending on the exposure time and plate-solve latency. Here is the flow at a glance:

[STEP 1] PLATE SOLVE Camera captures image → ASTAP resolves position │ Output: solved RA/Dec coordinates ▼ [STEP 2] KALMAN FILTER Python, in memory — NO communication with OnStep │ Input: measured error from plate solve │ Output: predicted drift + optimal rate ▼ [STEP 3] PYTHON CONVERTS Transforms predictions into OnStep commands │ Method A: SYNC (position correction) │ Method B: CUSTOM RATES (drift compensation) ▼ [STEP 4] ASCOM ALPACA HTTP PUT requests to telescope endpoint │ URL: http://IP:11111/api/v1/telescope/0/... ▼ [STEP 5] ONSTEP APPLIES Firmware receives commands, adjusts tracking Response: "1" (success)

Step 1: Plate Solve — Measuring the Real Error

The cycle begins with a camera exposure. The guide camera captures a 1-second image of the star field currently in the telescope's field of view. This image is handed to ASTAP, which performs plate-solving: matching the observed star patterns against its catalog to determine the exact Right Ascension and Declination the telescope is actually pointing at — not where it thinks it is, but where it truly is.

The plate-solve output produces two critical values: the solved coordinates (actual RA and Dec) and the measured error (the difference between the target position and the actual position, converted to arcseconds). This error is the raw material that feeds every subsequent step.

image = camera.capture_image(exposure=1.0)
solved = plate_solver.solve(image)
# → solved = {'ra': 0.7125, 'dec': 41.270}

target_ra = 0.712
target_dec = 41.269

ra_error = (target_ra - solved['ra']) * 3600   # Convert to arcsec
dec_error = (target_dec - solved['dec']) * 3600
# → ra_error = -0.45 arcsec, dec_error = +0.32 arcsec

At this stage, we have a measured error in arcseconds. The telescope is off by -0.45 arcsec in Right Ascension and +0.32 arcsec in Declination. These numbers are passed directly to the Kalman filter.

Step 2: Kalman Filter — The Mathematical Engine

The Kalman filter does not communicate with OnStep. It performs pure mathematical computation in Python memory — updating, predicting, and calculating optimal rates. Key architectural insight

This is the most misunderstood part of the pipeline. The Kalman filter runs entirely in Python memory. It does not open sockets, does not send HTTP requests, does not touch ASCOM. It is a pure mathematical function: given a measurement error and a time delta, update the internal state and compute the optimal tracking rate.

The state vector is four elements: RA position error, RA velocity (drift rate), Dec position error, and Dec velocity. On each cycle, three operations execute in sequence — update, predict, and compute optimal rate.

Operation 1: Update — Absorb the Measurement

# Kalman receives the measured error from plate solve
kalman_filter.update(
    measurement=(ra_error, dec_error),  # (-0.45, +0.32) arcsec
    dt=5.0                               # 5 seconds since last cycle
)

# Kalman state after update:
# x = [-0.45, -0.09, +0.32, +0.07]
#       RA     RA      Dec    Dec
#       pos   vel     pos    vel
# P = covariance matrix (uncertainty estimates)

The update step fuses the new measurement with the filter's prior prediction. The Kalman gain matrix K determines how much weight to give the new measurement versus the existing state estimate. A high-confidence measurement pulls the state toward the observed error; a noisy measurement is trusted less.

Operation 2: Predict — Forecast Future Drift

# Kalman predicts where the error will be in 5 seconds
predicted_ra_error, predicted_dec_error = kalman_filter.predict(dt=5.0)
# → "In 5 seconds, error will be ~-0.9 arcsec RA, +0.67 arcsec Dec"

The predict step projects the current state forward in time using the velocity estimates. If RA is drifting at -0.09 arcsec/sec, in 5 seconds the error grows to -0.45 + (-0.09 × 5) = -0.9 arcsec. This prediction is what makes Kalman more powerful than simple proportional correction — it anticipates where the error is going, not just where it is.

Operation 3: Compute Optimal Rate

# Kalman calculates the optimal tracking rate
optimal_ra_rate, optimal_dec_rate = kalman_filter.get_optimal_tracking_rate(
    sidereal_rate_ra=15.041067,   # Standard sidereal rate (arcsec/sec)
    sidereal_rate_dec=0.0,        # No Dec motion normally
    correction_gain=0.5           # 50% aggressiveness
)

# Formula used internally:
# optimal_ra = sidereal - (drift_velocity × gain) + (correction × gain)
# optimal_ra = 15.041067 - (-0.09 × 0.5) + (correction × 0.5)
# optimal_ra = 15.041067 + 0.045 + 0.023
# optimal_ra = 15.109 arcsec/sec

The optimal rate combines three factors: the base sidereal rate, the Kalman-predicted drift compensation, and a proportional correction based on the current error. The correction gain parameter (0.5 = 50%) controls how aggressively the system responds — higher values correct faster but risk oscillation.

Step 3: Python Converts to OnStep Commands

Now the critical translation happens. The Kalman filter has computed an optimal tracking rate — a single number in arcsec/sec. Python must convert this into two distinct types of commands that OnStep understands. These are not alternatives; both are sent every cycle, in a specific order.

Method A: SYNC

ascom.sync_to_coordinates(ra, dec)

Tells OnStep: "You are at this position NOW." Corrects the position error. Realigns the pointing model. Instantaneous effect — fixes where the telescope thinks it is.

Method B: CUSTOM RATES

ascom.command_string(":SXTR,offset#")

Tells OnStep: "Track at THIS speed." Compensates predicted drift. Continuous effect — adjusts how fast the telescope moves between plate solves.

Method A: SYNC — Position Correction

# After plate solve, always SYNC first
ascom.sync_to_coordinates(solved_ra=0.7125, dec_degrees=41.270)

# What happens internally:
# ASCOM HTTP PUT → http://192.168.1.100:11111/api/v1/telescope/0/synctocoordinates
# Body: RightAscension=0.7125  Declination=41.270  ClientID=1
# OnStep receives and:
#   1. Records the offset (pointing error)
#   2. Updates its internal pointing model
#   3. Future commands compensate automatically

The SYNC command is a standard ASCOM operation. It does not use Kalman predictions at all — it simply tells OnStep the real coordinates. OnStep then recalculates its pointing model, correcting systematic errors like polar alignment drift.

Method B: Custom Rates — Drift Compensation

# Calculate offset from sidereal rate
sidereal_ra = 15.041067  # arcsec/sec
ra_offset = optimal_ra_rate - sidereal_ra
# → ra_offset = 15.109 - 15.041067 = +0.068 arcsec/sec

dec_offset = optimal_dec_rate - 0.0
# → dec_offset = +0.05 arcsec/sec

# Send to OnStep via ASCOM
ascom.command_string(f":SXTR,{ra_offset:.6f}#")   # RA offset
ascom.command_string(f":SXTD,{dec_offset:.6f}#")  # DEC offset

This is where Kalman predictions become real motor commands. The :SXTR and :SXTD commands are OnStep-specific — they set RA and Dec tracking rate offsets relative to sidereal. A +0.068 arcsec/sec offset means OnStep tracks 0.068 arcsec/sec faster than sidereal, compensating the drift that Kalman predicted.

Step 4: ASCOM Alpaca — The HTTP Bridge

Every command from Python reaches OnStep through ASCOM Alpaca's HTTP REST interface. There is no serial connection, no direct USB link — it is standard HTTP PUT requests to a local network endpoint. This makes the system network-accessible and protocol-agnostic.

┌──────────────┐
│   KALMAN     │  (Python, in memory)
│  (Calculs)   │  optimal_ra = 15.109 arcsec/sec
└──────┬───────┘
       │  offset = optimal_ra - sidereal = +0.068
       ▼
┌──────────────┐
│   PYTHON     │  Build OnStep command
│              │  cmd = ":SXTR,+0.068#"
└──────┬───────┘
       │  ascom.command_string(cmd)
       ▼
┌──────────────┐
│ASCOM ALPACA  │  HTTP PUT
│  (Network)   │  → http://192.168.1.100:11111/api/v1/telescope/0/commandstring
└──────┬───────┘
       │  Body: Command=:SXTR,+0.068#&ClientID=1&ClientTransactionID=790
       ▼
┌──────────────┐
│ASCOM SERVER  │  Receives HTTP, parses command
│(192.168.1.100│  Converts to serial LX200 protocol
│  Port 11111) │
└──────┬───────┘
       │  Serial/USB: ":SXTR,+0.068#"
       ▼
┌──────────────┐
│   ONSTEP     │  Parses LX200 command
│ (Telescope)  │  Applies: tracking_rate_ra = 15.041067 + 0.068
│              │           = 15.109 arcsec/sec
└──────┬───────┘
       │  Response: "1" (success)
       ▼
┌──────────────┐
│   PYTHON     │  print("✓ Taux RA ajusté")
└──────────────┘

The ASCOM Alpaca server runs on the same machine as OnStep, listening on port 11111. Python sends a PUT request with the command string in the body. ASCOM translates the LX200 protocol command to serial, sends it to OnStep's USB interface, receives the response, and returns it to Python as a JSON object.

The Actual HTTP Exchange

# Python sends:
PUT http://192.168.1.100:11111/api/v1/telescope/0/commandstring
Content-Type: application/x-www-form-urlencoded

Command=:SXTR,+0.068#&ClientID=1&ClientTransactionID=790&Raw=true

# ASCOM server responds:
{
  "Value": "1",
  "ClientTransactionID": 790,
  "ServerTransactionID": 1234,
  "ErrorNumber": 0,
  "ErrorMessage": ""
}

The entire HTTP round-trip completes in approximately 50 milliseconds. Two additional PUT requests follow — one for the Dec offset, one for the SYNC coordinates — bringing the total ASCOM communication time to roughly 150 milliseconds for a complete correction cycle.

Step 5: OnStep Applies the Correction

OnStep receives the LX200 protocol commands and applies them immediately to its tracking motor controller. The :SXTR command modifies the RA tracking rate by adding the specified offset to the base sidereal rate. The :SXTD command does the same for Declination.

# OnStep receives and applies:
new_ra_rate = 15.041067 + 0.068 = 15.109 arcsec/sec
new_dec_rate = 0.0 + 0.050 = 0.050 arcsec/sec

# OnStep now tracks at these rates for the next 5 seconds
# until the next plate-solve cycle recalculates corrections

OnStep maintains these adjusted rates until the next correction arrives. Between plate-solve cycles, the telescope drifts at the corrected rate — which is why the Kalman prediction matters. A simple proportional correction would fix the current error but leave the drift uncompensated. Kalman's prediction of where the error will be in 5 seconds allows OnStep to track at exactly the right rate during the interval between corrections.

Real Timing — A Complete 5-Second Cycle

Here is the exact timeline of a single tracking cycle with timestamps, showing what happens at each moment:

T = 0 sec: CYCLE START ├─ [0.0–1.0s] Capture image (1-second exposure) ├─ [1.0–3.0s] Plate solve via ASTAP (2 seconds) ├─ [3.0s] Error measured: RA -0.45", Dec +0.32" │ ├─ [3.001s] KALMAN UPDATE │ kalman.update((-0.45, +0.32), dt=5.0) │ State updated internally │ ├─ [3.002s] KALMAN COMPUTES │ optimal_ra = 15.131 arcsec/sec │ optimal_dec = 0.05 arcsec/sec │ ├─ [3.003s] METHOD A: SYNC POSITION │ ascom.sync_to_coordinates(0.7125, 41.270) │ ↓ HTTP PUT (50ms) │ OnStep: "Position updated" ✓ │ ├─ [3.053s] METHOD B: CUSTOM RATES │ ascom.command_string(":SXTR,+0.090#") │ ↓ HTTP PUT (50ms) │ OnStep: "RA rate +0.09"/sec" ✓ │ │ ascom.command_string(":SXTD,+0.050#") │ ↓ HTTP PUT (50ms) │ OnStep: "Dec rate +0.05"/sec" ✓ │ ├─ [3.153s] CORRECTIONS APPLIED │ OnStep tracking at adjusted rates │ └─ [3.2–8.0s] OnStep compensates drift during wait Next plate solve arrives at T = 8 sec T = 8 sec: NEW CYCLE ├─ Plate solve → Error: RA -0.05", Dec +0.03" │ Much smaller than last cycle's -0.45", +0.32"! └─ Repeat...

The numbers tell the story. The first cycle measured -0.45 arcsec RA error. After Kalman corrections applied for 5 seconds, the next cycle measured only -0.05 arcsec. The system reduced error by 90% in a single cycle. With both SYNC and CUSTOM RATES working together, steady-state error settles to approximately ±0.5 arcseconds.

Why Both Methods Are Needed

The most common question about this architecture is: why use two methods instead of one? The answer lies in what each method corrects. SYNC fixes the current position. CUSTOM RATES fixes the rate of drift. They solve different problems, and both are essential.

Aspect SYNC (Method A) Custom Rates (Method B)
Corrects Current position error Future drift rate
Effect Realigns telescope Adjusts tracking speed
Duration Instantaneous Continuous (5–15 sec)
Uses Kalman No (just coordinates) Yes (predictions + rates)
ASCOM command synctocoordinates commandstring
OnStep command Standard ASCOM :SXTR / :SXTD
Without it Error accumulates No drift compensation
SYNC says: "Telescope, you are HERE."  Custom Rates say: "Telescope, track at THIS speed." Together, they achieve ±0.5 arcsecond precision. Why both methods are complementary, not redundant

Consider what happens if you use only SYNC without custom rates. The position is corrected to zero error, but the mount continues tracking at the standard sidereal rate. Over the next 5 seconds, the same systematic drift that caused the original error rebuilds. By the next plate solve, the error has grown back to nearly the same value. SYNC alone is a reactive correction that fights the same battle every cycle.

Now consider custom rates without SYNC. The Kalman filter predicts drift and adjusts the tracking rate accordingly, but the initial position offset is never corrected. The mount tracks at a slightly different speed, but from the wrong starting point. Error decreases more slowly than it should.

Used together in the correct order — SYNC first to establish the correct position, then CUSTOM RATES to maintain it — the two methods create a tight feedback loop. SYNC resets the error to zero. Custom rates prevent it from growing. The result is steady-state tracking error of approximately ±0.5 arcseconds, sustained over hours of imaging.

The OnStep Command Reference

The system uses a specific set of OnStep LX200 protocol commands, each serving a distinct purpose in the correction pipeline:

Command Description Example Kalman Usage
:SXTR,offset# Set RA offset :SXTR,+0.045# Primary
:SXTD,offset# Set DEC offset :SXTD,+0.032# Primary
:SX9A,rate# Set absolute RA rate :SX9A,15.086# Alternative
:SX9B,rate# Set absolute DEC rate :SX9B,0.050# Alternative
:RT,mult# Set rate multiplier :RT,1.003# Less precise

The recommended commands are :SXTR and :SXTD (offsets) rather than :SX9A/:SX9B (absolute rates). Offset commands are additive — they adjust relative to the current rate — which means OnStep's own sidereal tracking remains active as the baseline. Absolute rate commands would override the sidereal baseline entirely, which can cause issues if the correction is momentarily interrupted.

Fallback When Kalman Is Unavailable

The system degrades gracefully when the Kalman filter is not available — for example, during initial startup before enough plate-solve measurements have accumulated, or if the kalman_filter module fails to import. In these cases, a simple proportional correction replaces Kalman's predictions:

if self.kalman_filter and KALMAN_AVAILABLE:
    # Kalman available — use intelligent predictions
    new_ra_rate, new_dec_rate = self.kalman_filter.get_optimal_tracking_rate(
        base_ra_rate, base_dec_rate, correction_gain=0.5
    )
else:
    # Fallback — simple proportional correction
    ra_correction = -ra_error / time_elapsed
    dec_correction = -dec_error / time_elapsed
    new_ra_rate = self.state.current_ra_rate + ra_correction
    new_dec_rate = self.state.current_dec_rate + dec_correction

The proportional fallback still sends both SYNC and CUSTOM RATES — the two-method approach is maintained regardless of whether Kalman is active. The difference is that the fallback reacts to the current error only, without predicting future drift. This yields roughly ±2 arcsecond precision compared to Kalman's ±0.5 arcsecond.

Key Takeaways

1. Kalman Never Talks to OnStep

The filter is a pure mathematical function. It receives measurements and returns predictions. The translation to hardware commands happens in a separate Python layer.

2. Two Methods, Always Together

SYNC corrects position. Custom rates correct drift. Both execute every cycle, always in the same order: SYNC first, then rates. The order matters — rates must be applied to the correct position.

3. ASCOM Alpaca Is Just HTTP

Every command travels as an HTTP PUT request to a local network endpoint. There is no proprietary protocol, no direct hardware link. The ASCOM server translates HTTP to serial LX200 protocol for OnStep.

4. The Timing Budget Is Tight

Plate solve: 2 seconds. Kalman computation: 1 millisecond. Three HTTP requests: 150 milliseconds. OnStep application: instantaneous. Total correction time: approximately 2.2 seconds within a 5-second cycle.

5. Graceful Degradation Preserved

If Kalman is unavailable, the system falls back to proportional correction. If ASCOM is unreachable, the tracking loop continues with sidereal-only tracking. The mount never stops moving.
