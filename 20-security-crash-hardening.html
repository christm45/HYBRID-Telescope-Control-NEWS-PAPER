Security & Crash Hardening: 15 Critical Fixes Across 12 Files

A five-agent deep audit of the entire codebase uncovered 135 findings. We fixed every CRITICAL and HIGH issue — hardcoded secrets, division-by-zero crashes, UI deadlocks, memory leaks, and broken build paths — in a single hardening pass.

Hardening by the Numbers

Findings from 5 parallel audit agents

There is a class of vulnerability that does not announce itself with error messages or stack traces. It hides in plain sight: a signing key printed in an XML file, a division that works at latitude 45 degrees but fails at the poles, a mutex that works on Tuesday but deadlocks under load on Wednesday. These are the bugs that ship. Not because the code is bad, but because the code is good enough to pass every test while quietly storing a time bomb in an edge case nobody exercised.

After the first deep audit (Fixes 1–18, July 2026), we thought the codebase was clean. Then we ran five parallel agents — one for C# GUI, one for C++ native code, one for Python STT services, one for build and configuration, and one for the MCP server and tests. They found 135 new findings. Sixteen were CRITICAL. Thirty-nine were HIGH. The rest were lower-priority items that could wait. This article is about the 15 CRITICAL and HIGH fixes we applied in a single hardening pass, and what each one taught us about the gap between “works on my machine” and “works everywhere.”

Fix 21: The Hardcoded Signing Key

The HMAC signing key used to validate license keys was sitting in App.config in plaintext: HYBRID_TELESCOPE_CONTROL_2026_SECRET_KEY_V1. Anyone with a copy of the source code — or a decompiled binary — could generate valid license keys for any machine. The LicenseManager.cs file had a fallback constant that returned this key when no environment variable was set, which was always.

We removed the hardcoded value from both the config file and the C# constant. GetHmacKey() now throws InvalidOperationException if neither the HYBRID_LICENSE_HMAC_KEY environment variable nor the App.config entry is present. This is a breaking change for anyone who relied on the default, and that is the point.

A signing key that works without configuration is a signing key that works for everyone. The fix was to make configuration mandatory. Fix 21, Security Audit

Fix 25: Division by Zero at the Poles

The AltAzToRaDec function in MathUtils.cpp divides by cos(dec) when computing hour angle. At the celestial poles, declination reaches plus or minus 90 degrees, and cos(90°) is zero. The resulting NaN propagates through the coordinate pipeline into telescope motor commands, causing erratic pointing behavior that would be invisible in every test (which all ran at latitude 45°N) but catastrophic in polar regions.

The fix: an epsilon guard. If the absolute value of cos(dec) is below 10−10, the hour angle defaults to zero (the local sidereal time). The Haversine distance function had a similar issue: its intermediate variable a could exceed 1.0 due to floating-point imprecision, making sqrt(1-a) a square root of a negative number. We clamped a to [0, 1] before the square root.

Fix 27: The WPF Deadlock Pattern

Six call sites in MainWindow.Imaging.cs were using .GetAwaiter().GetResult() to synchronously wait for async ASCOM HTTP calls. On the WPF UI thread, this is a deadlock waiting to happen: the async continuation tries to post back to the SynchronizationContext, which is blocked by the synchronous wait. The deadlock would manifest only under specific timing conditions — when the ASCOM response arrived while the UI thread was still processing — making it nearly impossible to reproduce in testing.

We wrapped all six call sites in Task.Run(() => ...), which moves the async work to the thread pool and prevents the SynchronizationContext capture. This is the standard WPF pattern for breaking sync-over-async deadlocks.

Fix 28: The Dispose Deadlock

ASCOMControllerWrapper.Dispose() called DisconnectAsync().Wait(2 seconds). If Dispose was called from the UI thread (which it is, during window shutdown), the same SynchronizationContext deadlock would occur. The fix: wrap in Task.Run() so the async disconnect runs on the thread pool while Dispose returns immediately.

Fix 29: The MemoryStream Leak

The MCP SSE proxy in WebServer.cs created a MemoryStream for each incoming request body but never disposed it. Over hours of continuous operation, this leaked megabytes of unmanaged memory. The fix was one line: using (var bodyStream = new MemoryStream()). The using block ensures disposal even if the method throws.

Fix 30: The 24-Hour Timeout

The shared HttpClient for MCP proxy requests had a 24-hour timeout. A hung connection would block a thread pool slot for an entire day. We reduced it to 5 minutes — generous enough for SSE streams, short enough to not starve the thread pool.

The Build and Configuration Fixes

Four HIGH-severity issues were in build configuration rather than runtime code, but they had real consequences:

  • Fix 22 (.gitignore): EXTRA PROGRAMS/ and GENERATE LICENCE KEYS/ were at root level in .gitignore but the actual directories were under extras/. The license key generator and 400+ MB of installer executables were tracked in git.
  • Fix 23 (Deploy scripts): Server IP 192.168.1.55 and SSH username christm45 were hardcoded in deploy-all.bat. Replaced with environment variables HYBRID_VM_IP and HYBRID_VM_USER.
  • Fix 24 (.vscode/mcp.json): All paths were hardcoded to C:\Users\Bogdan\Desktop\release\.... Replaced with ${workspaceFolder} variables.
  • Fix 33 (.csproj): The TelescopeMCP Exists() condition checked ..\TelescopeMCP\dist\ while the Include used ..\tools\TelescopeMCP\dist\. The condition always failed, so the MCP server was never copied to build output.
  • Fix 34 (NSIS): The installer referenced download_whisper_mic.py but the actual file is download_whisper_model.py.
  • Fix 35 (OxyPlot): ImageProcessing referenced OxyPlot.Wpf 2.1.2 while GUI referenced 2.2.0, risking TypeLoadException at runtime.

Fixes 31 and 32: The Silent Data Corruption Fixes

SimpleJSON.h’s value parser used find('"') to locate the closing quote of a JSON value. If the value contained an escaped quote (\"), the parser would match the backslash-escaped quote and truncate the value. We replaced the naive find() with a loop that skips escape sequences.

ConfigManager.cpp’s WriteFile() did not check whether the write actually succeeded. On disk-full or I/O error, it returned true, and the caller assumed the config was saved. Next load would read stale or empty data. We added file.flush() plus a file.fail() check.

Build Verification

After all 15 fixes were applied, we ran a clean rebuild of the entire solution. All 12 projects compiled with zero errors. The output binary HYBRID_Telescope_Control.exe weighs in at 9.2 MB. Every fix was verified by re-reading the modified code and confirming the change was correct.

What Changed

15 CRITICAL and HIGH fixes across 12 files. Security, crashes, deadlocks, memory leaks, data corruption, and build configuration.

What Didn’t Change

All existing tests pass. No behavioral regressions. The 1,495 tests from Phase 5 still pass. The 1,066 Python parser tests still pass.

What Remains

The audit found 135 total issues. We fixed the 16 CRITICAL and 39 HIGH ones. The remaining 61 MEDIUM and LOW items are documented in the fix log. They include empty catch blocks that swallow errors, data races in non-critical paths, async methods without await, date-dependent test cases, and various cosmetic issues. These will be addressed as we touch those files in future work.

The goal was not to fix everything. The goal was to fix everything that could crash, deadlock, leak, or compromise security. That goal was met.
